# Tài liệu Golang Module Boundary & Implementation Blueprint

## 1. Mục đích tài liệu

Tài liệu này đặc tả chi tiết **Golang Module Boundary & Implementation Blueprint** cho hệ thống backend bookstore đang planning hiện tại. Mục tiêu là chuyển toàn bộ những gì đã chốt ở tầng nghiệp vụ và kiến trúc — bao gồm SRD, ERD/physical schema, Redis key & TTL spec, Kafka event contract, State Transition Specification, ADR pack — thành một blueprint triển khai cấp codebase để team backend có thể bắt đầu implement mà không bị mơ hồ về:
- chia module như thế nào;
- module nào sở hữu aggregate nào;
- package nào được phụ thuộc package nào;
- orchestration nằm ở đâu;
- state machine nằm ở đâu;
- Redis, Kafka, PostgreSQL được đóng gói thế nào;
- handler, worker, scheduler, webhook được đi dây ra sao;
- transaction boundary được đặt ở đâu;
- lỗi thiết kế nào phải tránh ngay từ ngày đầu.

Tài liệu này được viết theo tư duy **master backend developer**, nghĩa là ưu tiên:
- correctness trước convenience;
- ownership trước reuse bừa bãi;
- explicit boundary trước shared utility mơ hồ;
- observability và recoverability ngay từ thiết kế ban đầu;
- đơn giản hóa evolution path để tương lai có thể tách service nếu business thực sự cần.

## 2. Bối cảnh hệ thống

Project hiện tại có các đặc tính đã chốt:
- Golang là ngôn ngữ backend chính;
- Gin là HTTP framework lớp delivery;
- PostgreSQL là canonical source of truth;
- Redis dùng cho cache, coordination, lock, quota, session, hot-path transient state;
- Kafka là backbone asynchronous cho outbox, event-driven side effects, notification, invoice export, payment reconcile, shipment sync, reporting feed;
- hệ thống triển khai theo hướng modular monolith, không phải microservices ở phase hiện tại;
- business rules gồm COD cho physical-only order, VND only, e-invoice export ra provider ngoài, membership access cho các sách được gắn cờ phù hợp, admin override, flash-sale stock guarding, job deduplication.

## 3. Tư duy thiết kế tổng thể của blueprint

### 3.1 Triết lý chính

Blueprint này không sử dụng “clean architecture textbook” một cách giáo điều. Thay vào đó, nó áp dụng một biến thể thực dụng phù hợp với Golang modular monolith:
- tách rõ **domain**, **application**, **infrastructure**, **interfaces**;
- mỗi module có ownership rõ cho aggregate của mình;
- orchestration đi qua application service, không nhảy thẳng từ handler xuống repository theo kiểu CRUD trực tiếp;
- domain giữ business invariant và state transition policy;
- infrastructure chứa implementation phụ thuộc framework, database, Redis, Kafka, provider ngoài;
- interfaces chứa HTTP, webhook, jobs, consumers;
- module giao tiếp với nhau qua service contract nội bộ hoặc event contract, không import xuyên tường tùy tiện.

### 3.2 Tư duy master backend được áp dụng

Một backend developer mới thường mắc lỗi gom mọi thứ thành `service`, `repository`, `handler` rồi lớn dần thành codebase không rõ ownership. Tư duy master backend ở đây là:
- **Business change phải đi qua owner module (Nhà nào cửa nấy)**: Trạng thái của Đơn hàng (Order) chỉ có Order Service được phép sửa. Các service khác muốn đổi trạng thái thì phải "gõ cửa xin phép" (gọi API), không được leo rào vào tự sửa;
- **Read model và Write model không lẫn lộn (Sổ ghi chép khác Bảng tin):** Dữ liệu dùng để Validate/Lưu vào database (Write) tốn rất nhiều quy tắc khắt khe. Trong khi dữ liệu để hiển thị ra màn hình (Read) chỉ cần nhanh và gom từ nhiều nguồn. Bắt một Object gánh cả 2 việc sẽ tạo ra sự phình to và vô số các câu lệnh if-else;
- **shared package là nợ kiến trúc nếu lạm dụng**;
- **khả năng test được và reasoning được quan trọng hơn việc “ít file”**;
- **code structure phải phản ánh business boundaries**, không phản ánh framework.

## 4. Nguyên tắc kiến trúc bắt buộc

### 4.1 Nguyên tắc ownership
Mỗi aggregate (cụm dữ liệu) chỉ có đúng một chủ nhân. Muốn đổi dữ liệu? Vui lòng gửi công văn.
- Tư duy: Hãy coi Order Module là Phòng Kế Toán, Shipping Module là Phòng Giao Hàng.
- Cái két sắt chứa trạng thái Đơn Hàng nằm ở Phòng Kế Toán.
- Quy luật: Nhân viên Giao Hàng (Shipping) vừa giao xong món đồ, tuyệt đối không được tự chạy vào Kế Toán mở két sắt ra ghi dòng chữ "Đã giao xong". Làm vậy là vi phạm ranh giới.
- Cách làm đúng: Nhân viên Giao Hàng gửi một công văn (Call API - Contract) hoặc thông báo lên loa phường (Publish Kafka Event). Phòng Kế Toán nghe được sẽ tự lấy chìa khóa mở két của mình ra và ghi nhận trạng thái mới.

### 4.2 Nguyên tắc dependency direction

Dependency chỉ được chảy theo hướng:

```text
domain <- application <- interfaces
        <- infrastructure
```

Giải thích:
- `domain` Chứa thuần túy quy tắc kinh doanh (Ví dụ: "Hủy đơn phải hoàn tiền"). Lõi này hoàn toàn mù tịt về công nghệ. Không biết SQL, không biết Redis, không biết Framework Gin hay React là gì. Nhờ vậy, nghiệp vụ không bao giờ bị lỗi thời..
- `application` Làm đạo diễn điều phối. "Lấy data từ DB -> Ném cho Lõi tính toán -> Cất xuống DB".
- `infrastructure` implement các abstraction để kết nối DB, Redis, Kafka, provider ngoài.
- `interfaces` nhận input bên ngoài và gọi vào `application`.

### 4.3 Nguyên tắc transaction boundary
Giao dịch CSDL là phòng phẫu thuật đắt tiền. Đừng đưa bệnh nhân vào khi chưa sẵn sàng.

- Application (Đạo diễn): Là người được phép hô BEGIN TRANSACTION và COMMIT. Nó biết toàn bộ quy trình, nên nó mở và đóng giao dịch đúng lúc nhất.
- Domain (Bác sĩ chuyên môn): Chỉ quan tâm đến việc tính toán, đưa ra quyết định dựa trên dữ liệu có trên RAM. Không tự ý chạm vào DB.
- Interface/Handler (Lễ tân): Chuyên nhận HTTP request. Tuyệt đối cấm mở Transaction ở đây. Bạn không thể đưa bệnh nhân vào phòng mổ (giữ Database Connection) rồi bắt bác sĩ đứng đợi 3 giây để gọi API sang ngân hàng thanh toán thẻ. Cạn kiệt Connection Pool là điều chắc chắn xảy ra!.

### 4.4 Nguyên tắc event boundary - Ranh giới Sự kiện
Làm xong việc nhà mình thì bỏ thư vào hòm thư đi (Outbox). Đừng chạy sang nhà hàng xóm để sửa đồ

- Tư duy: Khi Order Module tạo đơn thành công, nó cần báo cho Inventory Module trừ kho.
- Nó sẽ ghi dữ liệu vào DB Order, đồng thời ghi 1 sự kiện ORDER_CREATED vào bảng Outbox trong cùng một Transaction của Database. (Đảm bảo dữ liệu và sự kiện đồng sinh cộng tử).
- Sau đó, một luồng chạy ngầm sẽ lấy thư trong bảng Outbox đi gửi (lên Kafka).
- Lưu ý sinh tử: Các service lắng nghe (Consumer) khi nhận được Event, chỉ được dùng logic của mình để sửa DB của mình. Tuyệt đối cấm Consumer dùng Repository chọc ngược lại DB của module khác

### 4.5 Nguyên tắc anti-corruption - Nguyên tắc Chống tham nhũng 
Trạm kiểm dịch biên giới: Không để ngôn ngữ ngoại bang làm ô nhiễm ngôn ngữ nội địa.

- Tư duy: Hệ thống của bạn kết nối với cổng thanh toán Stripe, Momo, VNPay. Mỗi bên có một kiểu định dạng dữ liệu (JSON) và mã lỗi khác nhau (Stripe lỗi card_declined, Momo lỗi 1001)
- Lỗi sai (Tham nhũng data): Bạn để nguyên cái mã lỗi card_declined của Stripe chạy tít sâu vào trong lõi Domain của mình để viết if-else. Hôm sau đổi cổng thanh toán sang PayPal, hệ thống của bạn vỡ nát
- Cách làm đúng: Phải có một "Adapter" (Người phiên dịch) đứng ở biên giới (tầng Infrastructure). Nó nhận mã lỗi của Stripe, dịch thành mã lỗi chuẩn của công ty bạn (Ví dụ: PAYMENT_INSUFFICIENT_FUNDS). Lõi (Domain) của bạn chỉ nói duy nhất ngôn ngữ của công ty bạn. Thế giới bên ngoài thay đổi thế nào thì lõi vẫn bình yên vô sự.

### 8.2 Ý nghĩa của từng thư mục

- `domain/entity`: mô hình aggregate và invariant gần dữ liệu nghiệp vụ nhất.
- `domain/valueobject`: enum state, money wrapper, typed ID, rule-bound values.
- `domain/policy`: state machine, guard, transition logic, invariant rules.
- `domain/event`: domain events nội bộ và mapping sang outbox event.
- `domain/repository.go`: interface mà domain/application cần.
- `application/command`: use cases có mutation.
- `application/query`: read use cases.
- `application/service`: orchestration và transaction boundary.
- `application/ports`: abstraction gọi sang module khác hoặc hạ tầng ngoài.
- `infrastructure/persistence`: implementation cụ thể cho PostgreSQL.
- `interfaces/http`: Gin handlers và mapping request/response.
- `interfaces/consumer`: Kafka consumer entrypoints của module.
- `interfaces/jobs`: scheduler entrypoints.
- `interfaces/webhook`: entrypoint từ provider ngoài nếu module có.

### 8.3 Tư duy master backend

Người mới thường đặt `handler -> service -> repository` cho mọi thứ và nghĩ vậy là đủ. Tư duy master backend là tách rõ:
- **điều gì là luật business**;
- **điều gì là orchestration**;
- **điều gì là hạ tầng**;
- **điều gì là adapter biên ngoài**.

Việc tách này giúp khi logic đổi, bạn sửa đúng chỗ và ít kéo theo side effect ngoài ý muốn.

## 9. Quy tắc dependency giữa các module

### 9.1 Giao tiếp được phép

Giao tiếp giữa module chỉ được theo các cách sau:
- gọi **application port** nội bộ đã được publish rõ ràng;
- publish/consume **Kafka event**;
- query read model hoặc query service chuyên biệt nếu chỉ cần đọc dữ liệu;
- dùng shared primitive thật sự generic.

### 9.2 Giao tiếp bị cấm

- import repository implementation của module khác;
- gọi trực tiếp vào `infrastructure/persistence` của module khác;
- sửa canonical table của module khác;
- dùng chung một package `utils/business` chứa logic của nhiều domain;
- lôi DTO HTTP của module này sang module khác dùng như internal contract.

### 9.3 Ví dụ dependency đúng

- `order` gọi `payment` qua `PaymentPort` để khởi tạo payment.
- `reader` gọi `entitlement` qua `EntitlementQueryPort` để check quyền.
- `shipment` publish event, `order` consume outcome để cập nhật summary state.
- `membership` publish event, `entitlement` consume để expire/revoke rights liên quan.

### 9.4 Tư duy master backend

Một codebase monolith chết không phải vì có quá nhiều file, mà vì **mọi thứ biết quá nhiều về nhau**. Boundary tốt làm giảm knowledge leakage. Đây là thứ FE developer thường ít bị đau vì UI component ít khi sở hữu canonical business state; backend thì ngược lại.

## 10. Thiết kế layer chuẩn

## 10.1 Domain layer

### Trách nhiệm
- giữ entity, value object, aggregate behavior;
- giữ state enum và transition policy;
- giữ business invariant và validation cốt lõi;
- phát sinh domain event nội bộ.

### Không được làm
- không dùng SQL/Redis/Kafka client;
- không parse HTTP request;
- không phụ thuộc framework;
- không gọi network ra ngoài.

### Ví dụ domain method tốt
- `Order.CanTransition(trigger, ctx)`
- `Order.ApplyPaymentCaptured(...)`
- `Membership.Expire(now)`
- `Entitlement.Revoke(reason)`

### Tư duy master backend

Domain layer phải là nơi bạn có thể đọc và hiểu “business nghĩ gì” mà không bị tiếng ồn từ framework, logging, JSON hay SQL làm loãng đi.

## 10.2 Application layer

### Trách nhiệm
- orchestration use case;
- transaction boundary;
- gọi repository/ports;
- phối hợp nhiều aggregate/module để hoàn tất một use case;
- map input/output với DTO nội bộ.

### Không được làm
- không nhúng SQL raw trong service;
- không encode business rule rải rác thay domain policy;
- không tự quyết canonical state của module khác.

### Ví dụ use cases
- `CreateOnlineOrder`
- `MarkPaymentCaptured`
- `ApproveRefund`
- `ActivateMembershipAfterPaidOrder`
- `IssueDownloadToken`

### Tư duy master backend

Application layer là nơi điều phối “ai làm gì trước/sau”, còn domain là nơi quyết định “có được phép không”. Nhiều codebase hỏng vì nhét cả hai thứ vào một `service.go` khổng lồ.

## 10.3 Infrastructure layer

### Trách nhiệm
- PostgreSQL repository;
- Redis coordination/cache/quota implementation;
- Kafka producer/consumer adapter;
- provider adapters cho payment, carrier, invoice;
- file storage, mail provider, secrets, external SDK.

### Không được làm
- không tự thêm business rule ngoài domain/application contract;
- không tự mutate state trái flow khi adapter nhận external callback lạ;
- không trả thẳng provider model ra domain nếu chưa anti-corruption.

### Tư duy master backend

Infrastructure là “thế giới bẩn” nhưng phải được nhốt đúng chỗ. Nếu để nó chảy ngược lên domain, về lâu dài business rule sẽ bị bóp méo theo cách provider ngoài hoạt động.

## 10.4 Interfaces layer

### Trách nhiệm
- HTTP handlers;
- webhook handlers;
- Kafka consumers;
- scheduler jobs/CLI entrypoints;
- request/response mapping;
- auth context extraction, correlation/tracing injection.

### Không được làm
- không mở transaction business lớn ngay trong handler;
- không tự viết logic state machine;
- không query chéo module bừa bãi để “tiện response”.

### Tư duy master backend

Handler tốt là adapter mỏng. Nếu handler dài 300 dòng và hiểu quá rõ business, codebase đang rò rỉ boundary nghiêm trọng.

## 12. Query side và read model

### 12.1 Nguyên tắc tách query khỏi command

Không nhất thiết áp dụng CQRS cứng, nhưng blueprint này khuyến nghị tách:
- command use case: mutate state, enforce invariant;
- query use case: tối ưu đọc, có thể join nhiều bảng hoặc dùng cache.

### 12.2 Query repository

Mỗi module có thể có:
- `Repository` cho aggregate canonical write/read close-to-domain;
- `QueryRepository` cho màn hình/list/report/detail optimized.

Ví dụ:
- `order.domain.Repository` để load/save aggregate;
- `order.application.query.OrderQueryRepository` để list orders cho admin với filter, pagination, join summary.

### 12.3 Tư duy master backend

Người mới hay cố ép một repository làm tất cả. Backend giỏi chấp nhận rằng read model và write model có ưu tiên khác nhau; tách vừa đủ sẽ giảm phức tạp hơn là cố “generic hóa” một interface khổng lồ.

## 14. Transaction design blueprint

## 14.1 Quy tắc chung

- Một use case mutation nên cố gắng chỉ có **một transaction business chính** trên PostgreSQL.
- Trong transaction đó có thể:
  - load aggregate;
  - validate;
  - apply transition;
  - persist aggregate changes;
  - insert outbox events;
  - insert audit logs nếu policy yêu cầu.

- Không gọi network ra payment/carrier/mail provider bên trong transaction DB dài nếu không cần thiết.

## 14.2 Ví dụ transaction: `MarkPaymentCaptured`

Flow đúng:
1. webhook handler parse + verify;
2. gọi application use case `HandlePaymentCapturedWebhook`;
3. mở DB transaction;
4. lock payment/order cần thiết;
5. check idempotency;
6. transition payment -> captured;
7. transition order -> paid;
8. insert outbox events;
9. commit;
10. async consumers xử lý membership/entitlement/notification/invoice.

### Tư duy master backend

Network I/O ngoài transaction là tư duy quan trọng. Người mới hay “gộp hết cho chắc”, nhưng transaction giữ quá lâu sẽ tăng contention và kéo theo nhiều failure mode khó lường.

## 15. Event publishing & consumer blueprint

## 15.1 Outbox integration

Mỗi module owner khi commit mutation quan trọng phải:
- tạo domain events nội bộ;
- map domain events -> outbox records;
- insert outbox trong cùng transaction.

Pseudo structure:

```go
type DomainEvent interface {
    EventType() string
    AggregateType() string
    AggregateID() int64
    Payload() any
}
```

## 15.2 Consumer organization

Consumer không đặt chung vào một thư mục “kafka” khổng lồ. Thay vào đó:
- mỗi module có consumer entrypoints của riêng mình;
- runner chung nằm ở `platform/kafka`;
- routing event tới consumer của module owner hoặc module subscriber.

Ví dụ:
- `membership/interfaces/consumer/order_paid_consumer.go`
- `entitlement/interfaces/consumer/membership_events_consumer.go`
- `notification/interfaces/consumer/domain_email_bridge.go`

## 15.3 Tư duy master backend

Đặt consumer theo module giúp side effect ownership rõ ràng. Nếu tất cả consumer nằm chung một nồi, rất nhanh bạn sẽ không biết “ai thực sự sở hữu phản ứng đối với event này”.

## 21. Interface design guidelines

### 21.1 Repository interfaces

Repository interface nên vừa đủ cho use cases của module, không generic quá mức.

Ví dụ tốt:

```go
type OrderRepository interface {
    GetByIDForUpdate(ctx context.Context, id int64) (*entity.Order, error)
    Save(ctx context.Context, order *entity.Order) error
}
```

Ví dụ xấu:

```go
type Repository[T any] interface {
    Create(ctx context.Context, v T) error
    Update(ctx context.Context, v T) error
    Delete(ctx context.Context, id int64) error
    Find(ctx context.Context, id int64) (T, error)
}
```

Lý do xấu:
- generic CRUD abstraction thường xóa mất semantics nghiệp vụ;
- khó diễn tả locking, consistency requirements, aggregate-specific operations.

### 21.2 Port interfaces giữa module

Port nên mô tả **capability**, không mô tả chi tiết persistence.

Ví dụ:
- `CheckEntitlement(ctx, userID, bookID)`
- `CreatePaymentIntent(ctx, orderSnapshot)`
- `ReserveStock(ctx, reservationRequest)`

### 21.3 Tư duy master backend

Interface không phải để “decouple càng nhiều càng tốt”, mà để **giấu đúng thứ nên giấu**. Interface vô nghĩa sẽ làm code khó đọc hơn chứ không modular hơn.

## 22. Error handling blueprint

## 22.1 Phân loại lỗi

| Nhóm lỗi | Ví dụ | Mapping |
|---|---|---|
| Business rule violation | COD không hợp lệ, entitlement expired | 4xx domain error |
| Idempotency conflict | duplicate checkout, duplicate callback processing | 409/200 idempotent replay tùy case |
| Infra transient | Redis timeout, Kafka timeout, network fail | retry / 5xx |
| External permanent | provider payload invalid | 4xx/park/DLQ |
| Internal bug | nil pointer, invariant broken | 5xx + alert |

## 22.2 Domain error package

Mỗi module nên có domain errors rõ:
- `ErrIllegalTransition`
- `ErrOrderNotCancelable`
- `ErrPaymentAlreadyCaptured`
- `ErrMembershipExpired`
- `ErrConcurrentReaderLimitExceeded`

### Tư duy master backend

Business lỗi mà bị trả thành `500 internal error` là dấu hiệu backend chưa trưởng thành. Error taxonomy tốt giúp API, logs, retry logic và support team nói cùng một ngôn ngữ.

- HTTP request -> transaction -> outbox -> consumer -> email/invoice/payment side effect.

### Tư duy master backend

Observability không phải “có log là được”. Nó là khả năng trả lời trong production: chuyện gì xảy ra, ở đâu, theo thứ tự nào, và ảnh hưởng tới ai.

## 24. Migration & schema change blueprint

## 24.1 Nguyên tắc

- migration phải backward-compatible nếu deploy rolling;
- không drop column/constraint ngay khi code mới vừa ra;
- dùng expand-and-contract khi cần;
- migration độc lập với app startup ở production là tốt hơn.

## 24.2 Thứ tự an toàn

1. add columns/tables/indexes mới;
2. deploy code ghi cả dữ liệu mới nếu cần;
3. backfill;
4. chuyển read path;
5. dọn dữ liệu cũ ở release sau.

### Tư duy master backend

Schema change là một phần của system design, không phải việc DBA làm sau. Rất nhiều lỗi production đến từ deployment sequence sai chứ không phải từ business logic sai.

## 25. Testing blueprint

## 25.1 Test pyramid thực dụng

- Unit test cho domain policy/state transitions;
- Integration test cho repository + transaction + Redis script;
- Contract test cho OpenAPI và Kafka schema;
- End-to-end smoke cho flow checkout-payment-entitlement-download;
- Concurrency test cho inventory/payment/download/reader.

## 25.2 Vị trí test

```text
test/
├── integration/
│   ├── order/
│   ├── payment/
│   ├── entitlement/
│   └── inventory/
├── e2e/

```

### Tư duy master backend

Đừng chỉ test handler response. Những bug tốn tiền nhất thường nằm ở race condition, duplicate event, retry flow và transaction boundary.

## 26. Security blueprint

## 26.1 Phân lớp bảo mật

- AuthN: session/token validation;
- AuthZ: RBAC và ownership checks;
- Transport: TLS, secure headers, provider webhook signature verify;
- Data: mask PII, secret management, audit logs;
- Abuse control: rate limiting, download quota, login throttling.

## 26.2 Quy tắc code

- không log secret/token thô;
- không log full provider payload nếu có dữ liệu nhạy cảm;
- secret đọc qua config manager;
- admin endpoints phải có permission check rõ ràng.

### Tư duy master backend

Security không phải một middleware. Nó là tập quyết định xuyên suốt từ model dữ liệu, log policy, webhook verification đến quyền admin.

## 27. Composition root blueprint

## 27.1 `cmd/api/main.go`

Nên làm:
- load config;
- init logger/tracing;
- init DB/Redis/Kafka producer;
- build module graph;
- register routes/middlewares;
- start HTTP server.

Không nên làm:
- business logic;
- raw SQL;
- provider-specific branching.

## 27.2 `cmd/worker/main.go`

Nên làm:
- init consumer runner;
- register module consumers;
- register outbox publisher;
- run graceful shutdown.

## 27.3 `cmd/scheduler/main.go`

Nên làm:
- init scheduler registry;
- đăng ký periodic jobs theo config;
- lock leader/single-run guard nếu cần.

### Tư duy master backend

Composition root là nơi duy nhất được “biết hết” các implementation cụ thể. Nếu module business bắt đầu tự new repository/client bên trong, dependency graph sẽ sớm hỗn loạn.

## 28. Wiring blueprint ví dụ

## 28.1 Order module wiring

```go
orderRepo := orderpg.NewRepository(db)
orderQueryRepo := orderpg.NewQueryRepository(db)
auditPort := auditapp.NewPort(auditRepo)
outboxPort := outboxapp.NewPort(outboxRepo)
paymentPort := paymentapp.NewInternalPort(paymentService)
inventoryPort := inventoryapp.NewInternalPort(inventoryService)
orderService := orderapp.NewService(txManager, orderRepo, orderQueryRepo, paymentPort, inventoryPort, auditPort, outboxPort, clock)
orderHandler := orderhttp.NewHandler(orderService)
```

### Tư duy thiết kế

- wiring ở composition root;
- module `order` phụ thuộc abstraction/capability, không phụ thuộc implementation chi tiết của `payment` hay `inventory`;
- dễ test bằng mock/fake ports ở application layer.

## 29. Danh mục anti-pattern cần tránh

### 29.1 Architectural anti-patterns

- shared `service` package dùng chung toàn hệ thống;
- repository generic CRUD cho mọi aggregate;
- domain import trực tiếp `gin.Context`;
- business logic nằm trong SQL query string builder;
- consumer sửa state chéo module bằng raw SQL;
- sử dụng Redis như canonical store cho quyền/tài chính;
- hardcode event names rải rác;
- handler vừa validate vừa transition vừa publish event vừa gửi email.

### 29.2 Golang-specific anti-patterns

- package quá to và export tràn lan;
- dùng `interface{}`/`any` vô tội vạ cho DTO domain;
- context propagation không nhất quán;
- function signature 12 tham số thay vì request object;
- struct god-object ôm 20 dependency;
- file `utils.go` khổng lồ.

### 29.3 Tư duy master backend

Anti-pattern thường sinh ra từ “tiện trước mắt”. Master backend luôn trả lời câu hỏi: **sự tiện này đang vay nợ kiến trúc ở đâu, và ai sẽ trả sau này?**

## 31. Acceptance criteria

### 31.1 Boundary
- Mọi aggregate canonical có owner module rõ ràng.
- Không có module nào sửa canonical state của module khác bằng repository trực tiếp.
- Dependency direction không bị đảo chiều.

### 31.2 Implementation readiness
- Có thể tạo skeleton packages cho từng module ngay từ blueprint này.
- Có thể map use cases từ State Transition Spec sang application commands.
- Có thể map Kafka Event Contract sang outbox + consumers mà không mâu thuẫn.
- Có thể map Redis spec sang coordination/cache implementation trong từng module.

### 31.3 Operational readiness
- API, worker, scheduler có binary riêng.
- Có chỗ đặt observability, retry, DLQ, migration và runbooks.
- Có chiến lược test rõ cho logic, infra và concurrency.
