# Tài liệu OpenAPI Specification Chi Tiết cho Bookstore Backend

## 1. Mục đích tài liệu

Tài liệu này đặc tả chi tiết **OpenAPI Specifications** cho toàn bộ hệ thống backend bookstore đang planning hiện tại, bao gồm cả **public APIs** và **admin APIs**. Tài liệu được viết theo định hướng **design-first**, nghĩa là API contract phải được chốt rõ trước khi team bắt đầu implement, để bảo đảm đồng bộ giữa:

- yêu cầu nghiệp vụ trong SRD;
- state machine đã chốt;
- physical schema;
- Redis hot-path;
- Kafka event contract;
- Golang module boundary & implementation blueprint.

Tài liệu này không chỉ mô tả endpoint, mà còn chốt luôn:

- chuẩn versioning;
- auth/authz model;
- chuẩn request/response envelope;
- error model;
- pagination/filter/sort conventions;
- idempotency cho các endpoint nhạy cảm;
- retry semantics;
- security headers;
- trạng thái nghiệp vụ và side effect expectations;
- quy tắc thiết kế mang tư duy của **master backend developer**.

## 2. Phạm vi API

### 2.1 Public APIs

Public APIs phục vụ:

- người dùng cuối;
- web/mobile client;
- reader client;
- cart/checkout/payment flows;
- thư viện cá nhân, tải file, membership.

### 2.2 Admin APIs

Admin APIs phục vụ:

- back-office operations;
- quản trị catalog, price, inventory, shipment, refunds, membership, entitlements;
- vận hành order/payment;
- retry/recovery actions có kiểm soát;
- audit/monitoring read models.

### 2.3 Ngoài phạm vi tài liệu này

- provider-to-provider callback specs chi tiết theo vendor;
- internal RPC giữa modules vì hệ hiện tại là modular monolith;
- GraphQL;
- public partner APIs ở phase sau.

## 3. Tư duy thiết kế API của master backend

### 3.1 API không phản chiếu database

Một sai lầm phổ biến là thiết kế API theo bảng dữ liệu: `/books`, `/orders`, `/payments`, rồi bắt client tự ghép business flow. Tư duy backend thực chiến là API phải phản ánh **business capability** và **use case**, không phản ánh raw schema.

Ví dụ:

- `POST /checkout/orders` đúng hơn nhiều so với bắt FE tạo order rồi tự gọi init payment rồi tự patch trạng thái.
- `POST /reader/sessions` đúng hơn `PATCH /reader_session_status`.

### 3.2 API phải thể hiện ownership và invariant

API tốt phải làm rõ:

- endpoint nào có quyền mutate state gì;
- state nào chỉ được đọc;
- đâu là command endpoint, đâu là query endpoint;
- request nào bắt buộc `Idempotency-Key`;
- response nào chỉ là accepted chứ chưa phải final business result.

### 3.3 API không được “quá tiện” mà phá kiến trúc

Master backend developer không chấp nhận kiểu API “cho tiện FE” nhưng phá ownership, ví dụ:

- admin trực tiếp patch `payment_state=captured`;
- FE gửi `total_amount` để server tin;
- endpoint vừa tạo order vừa gửi email sync vừa finalize payment.

API chuẩn phải giữ correctness trước, rồi mới tối ưu DX.

## 4. OpenAPI meta chuẩn

### 4.1 OpenAPI version

Chuẩn tài liệu:

- OpenAPI `3.1.0`

### 4.2 Base servers

```yaml
servers:
  - url: https://api.bookstore.example.com
    description: Production
  - url: https://staging-api.bookstore.example.com
    description: Staging
  - url: http://localhost:8080
    description: Local
```

### 4.3 API versioning strategy

- Dùng path versioning: `/api/v1/...`
- Không đổi breaking change trong cùng major version nếu tránh được.
- Nếu breaking change lớn, tạo `/api/v2` thay vì lén sửa contract cũ.

### 4.4 Response format

- JSON là định dạng chuẩn cho toàn bộ API.
- Timestamp dùng ISO 8601 UTC.
- Số tiền trả ở dạng integer VND, không dùng float.

## 5. Chuẩn xác thực và phân quyền

## 5.1 Auth model

### Public APIs

- Bearer access token cho endpoint yêu cầu đăng nhập.
- Một số endpoint công khai như catalog/search/detail không yêu cầu auth.

### Admin APIs

- Bearer access token + RBAC permissions bắt buộc.
- Có thể bổ sung step-up verification cho hành động nhạy cảm sau này.

## 5.2 Security schemes mẫu

```yaml
components:
  securitySchemes:
    bearerAuth:
      type: http
      scheme: bearer
      bearerFormat: JWT
```

## 5.3 Permission philosophy

Admin không có nghĩa được patch mọi thứ. Permission phải theo capability:

- `catalog.read`
- `catalog.write`
- `order.read`
- `order.cancel`
- `refund.approve`
- `membership.revoke`
- `inventory.adjust`
- `shipment.override`
- `audit.read`

### Tư duy master backend

Phân quyền tốt không gắn với UI menu, mà gắn với **business action có khả năng thay đổi canonical state**.

## 7. Chuẩn pagination, filtering, sorting

## 7.1 Cursor pagination mặc định cho dữ liệu biến động

Áp dụng cho:

- order list
- payment list
- refund list
- download history
- notification history
- audit logs

Query params:

- `limit`
- `cursor`
- `sort`

Ví dụ:

```http
GET /api/v1/orders?limit=20&cursor=eyJpZCI6MTAwMjQ1fQ==&sort=-created_at
```

## 7.2 Offset pagination chỉ dùng cho danh mục nhỏ/ổn định

Áp dụng có chọn lọc cho:

- admin catalog facets nhỏ;
- lookup lists ít thay đổi.

## 7.3 Filtering conventions

- dùng query params phẳng, dễ đọc;
- không truyền object filter JSON trong query string;
- filter ngày dùng `created_from`, `created_to`;
- filter range tiền dùng `min_total_vnd`, `max_total_vnd`.

### Tư duy master backend

Cursor pagination phù hợp hơn cho dữ liệu thay đổi liên tục vì tránh các lỗi offset drift và thường scale tốt hơn trên tập lớn [web:131][web:134][web:137]. Offset đơn giản hơn nhưng không nên lạm dụng cho feed biến động mạnh [web:126].

## 8. Idempotency, retry, rate limiting

## 8.1 Endpoint bắt buộc `Idempotency-Key`

- `POST /api/v1/checkout/orders`
- `POST /api/v1/payments/{payment_id}/retry`
- `POST /api/v1/refunds`
- các admin commands nhạy cảm như revoke membership, manual inventory adjustment, retry invoice export

## 8.2 Header chuẩn

```http
Idempotency-Key: 7e8dbd7a-2b3d-4b9f-8ec2-a0aa11223344
```

## 8.3 Retry semantics

- Nếu request cũ đã hoàn tất thành công với cùng key và cùng payload: trả lại kết quả cũ.
- Nếu request cũ đang xử lý: trả `409 Conflict` hoặc `202 Accepted` theo use case.
- Nếu key trùng nhưng payload khác: trả lỗi `409 IDEMPOTENCY_KEY_REUSED_WITH_DIFFERENT_PAYLOAD`.

## 8.4 Rate limiting headers

Khi cần:

- `X-RateLimit-Limit`
- `X-RateLimit-Remaining`
- `Retry-After`

### Tư duy master backend

Idempotency là hợp đồng business, không chỉ là cơ chế kỹ thuật. Checkout hay refund mà không có idempotency thì production sớm muộn sẽ xuất hiện double order hoặc double side effect.

## 9. Chuẩn enum và field conventions

### 9.1 Chuẩn naming JSON

- Dùng `snake_case` thống nhất cho field JSON.
- ID luôn là integer hoặc string typed-ID rõ ràng, không mập mờ.
- Trường thời gian: `created_at`, `updated_at`, `paid_at`, `expires_at`.

### 9.2 Money convention

- `*_vnd` là integer.
- Không trả float.
- Currency hiện tại cố định `VND`, nhưng response vẫn có thể giữ field `currency` để future-proof.

### 9.3 State fields

- Field trạng thái canonical dùng hậu tố `_state` nếu cần rõ context: `order_state`, `payment_state`, `shipment_state`.
- Response tóm tắt có thể dùng `status` khi không gây mơ hồ.

## 10.4 Checkout & Order APIs

### 10.4.1 `POST /api/v1/checkout/orders`

**Mục đích**

- tạo order từ cart/current request.

**Headers**

- `Authorization: Bearer <token>`
- `Idempotency-Key: <uuid>`

**Request body**

```json
{
  "payment_method": "vnpay",
  "shipping_address_id": null,
  "billing_profile_id": null,
  "coupon_code": "SALE10",
  "items": [
    {
      "sellable_sku_id": 1302,
      "quantity": 1
    },

      "sellable_sku_id": 2002,

```

**Business rules**

- server luôn re-price;

**Response 201**

```json

  "data": {
    "order_id": 100245,
    "order_no": "ORD202604170001",
    "order_state": "pending_payment",
    "payment": {
      "payment_id": 77891,
      "payment_method": "vnpay",
      "payment_state": "initiated",
      "redirect_url": "https://pay.example.com/redirect-token"
    },
    "pricing": {
      "currency": "VND",
      "subtotal_vnd": 250000,
      "discount_vnd": 20000,
      "shipping_fee_vnd": 0,
      "tax_vnd": 0,
      "total_vnd": 230000

  "meta": {
    "request_id": "req_4",
    "trace_id": "trace_4"
```

## 13. HTTP status code guidelines

| Status                      | Khi dùng                                                             |
| --------------------------- | -------------------------------------------------------------------- |
| `200 OK`                    | đọc thành công, command hoàn tất sync                                |
| `201 Created`               | tạo mới resource hoặc command tạo aggregate mới                      |
| `202 Accepted`              | command đã nhận nhưng final result async                             |
| `204 No Content`            | xóa hoặc revoke session thành công không cần body                    |
| `400 Bad Request`           | query/body sai shape cơ bản                                          |
| `401 Unauthorized`          | chưa auth hoặc token không hợp lệ                                    |
| `403 Forbidden`             | không có quyền                                                       |
| `404 Not Found`             | resource không tồn tại hoặc không thuộc quyền user                   |
| `409 Conflict`              | idempotency conflict, illegal transition, duplicate stateful request |
| `410 Gone`                  | cursor/token không còn hợp lệ trong vài case                         |
| `422 Unprocessable Entity`  | business validation fail                                             |
| `429 Too Many Requests`     | rate limited                                                         |
| `500 Internal Server Error` | lỗi nội bộ không mong muốn                                           |
| `503 Service Unavailable`   | dependency tạm lỗi / maintenance                                     |

### Tư duy master backend

HTTP status là một phần của contract. Đừng dùng `200` cho mọi thứ rồi nhét trạng thái vào body; điều đó làm client mất khả năng retry và xử lý theo chuẩn [web:130][web:136][web:139].

## 17. Mapping API với event-driven side effects

Một số endpoint thành công không đồng nghĩa toàn bộ side effects đã hoàn thành ngay:

- `POST /checkout/orders` có thể trả order + payment intent, nhưng email/invoice/entitlement diễn ra async sau khi payment captured.
- `POST /refunds` chỉ tạo yêu cầu refund, chưa chắc refund hoàn tất.
- `POST /admin/invoices/exports/{id}/retry` chỉ enqueue retry flow.

### Tư duy master backend

API response phải trung thực về **mức độ hoàn tất**. Đừng trả thông tin khiến FE hiểu nhầm rằng business process đã hoàn tất trong khi thực tế mới chỉ vào hàng đợi.

## 19. Acceptance criteria

### 19.1 Về thiết kế

- Public và Admin APIs đều có boundary rõ.
- Command và query endpoints được tách tương đối rõ ràng.
- Không có endpoint nào cho phép patch canonical state một cách mơ hồ.
- Error model, pagination, idempotency và security headers đã thống nhất.

### 19.2 Về nghiệp vụ

- Checkout phản ánh đúng pricing authority của backend.
- Payment API không cho client finalize thủ công.
- Membership/entitlement/download/reader APIs bám đúng business-right model.
- Admin APIs hỗ trợ vận hành nhưng không phá state machine.

### 19.3 Về implementation

- Có thể tách file OpenAPI theo module như blueprint codebase.
- Có thể sinh server/client stubs nếu muốn.
- Có thể viết contract tests trực tiếp từ đặc tả này.

## 20. Hạng mục nên làm tiếp sau OpenAPI

Sau khi chốt OpenAPI Specs, các tài liệu nên làm tiếp là:

- Data Dictionary chi tiết cho toàn bộ schema;
- Test Strategy & QA Matrix;
- Failure Recovery Runbook;
- Deployment & Environment Blueprint;
- Coding Standards & Code Review Checklist cho Golang team.

Tài liệu OpenAPI này là bước chuyển từ kiến trúc và state machine sang contract tích hợp thực tế giữa backend và clients, đồng thời là công cụ cực quan trọng để giữ cho implementation không trôi khỏi nghiệp vụ khi dự án bắt đầu đi vào coding thật.
